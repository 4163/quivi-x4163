# Quivi Standalone WebP/GIF Support

## Quivi

Quivi is an image viewer (specialized for comic/manga reading) which supports many file formats and compressed (zip, rar) files. It is aimed for fast & easy file browsing with keyboard or mouse.

- Original project: [Quivi](http://quivi.sourceforge.net/)
- Later continuation/fork: [qazmlpok/quivi](https://github.com/qazmlpok/quivi)

## quivi-x4163

This folder is based on the standalone Quivi 1.2.1 build. The work focused on image format support and animated playback while keeping the original portable-folder style and view zoom/fit behavior intact.

This prepared project package was generated by OpenAI's Codex using GPT 5.5 at a Medium Reasoning level.

Implemented:

- Static `.webp` support through FreeImage.
- Animated `.gif` support.
- Animated `.webp` support.
- Bundled helper runtime for animated GIF/WebP frame extraction.
- Hidden helper execution, so no Python console window appears during playback.
- Fresh-machine behavior: animated GIF/WebP support does not require a system Python install.
- Unicode path handling fixes.
- Case-insensitive/symbolic `FitType` settings compatibility.
- View menu scaling options:
  - `None`
  - `Bicubic`
  - `Lanczos`
- Key-bindable scaling commands through the existing Options key binding system.

## Scaling Behavior

Default scaling is `Bicubic`.

That preserves Quivi's original behavior: before the menu was added, the FreeImage resize path used `FreeImage.FILTER_BICUBIC`.

Scaling options apply broadly to images displayed through Quivi's FreeImage path, not only to GIF/WebP files.

Current behavior:

- `None`: skips pre-resampling on Windows and lets GDI stretch directly. This is the fastest path and is useful for large animations.
- `Bicubic`: default/original Quivi behavior.
- `Lanczos`: higher quality but more expensive.

Thumbnails are separate and still use the existing thumbnail scaling path.

Future scaling work should aim to fix and optimize the resampling behavior used by these scaling methods. Bicubic remains the default, but large animated images can still become expensive when high-quality scaling is active, especially after zoom or fit changes.

## Animated Playback Design

Animated GIF/WebP files are predecoded into frames using the bundled Pillow helper, then loaded by Quivi.

The helper writes:

- `frame_0000.bmp`
- `frame_0001.bmp`
- ...
- `delays.txt`

Quivi reads the generated BMP frames and frame delays, then plays them through the existing canvas/rendering path.

The helper preference order is:

1. `webp_extract_pillow_runtime/webp_extract_pillow/webp_extract_pillow.exe`
2. `webp_extract_pillow.exe`
3. `webp_extract_pillow.py` through Python/Pythonw, for development fallback only

## Performance Notes

Several animation-performance fixes are included:

- Uses the PyInstaller `onedir` helper first to avoid one-file unpack/startup overhead.
- Starts playback only after the active displayed animated frame set has been preloaded.
- If an async preload is already running, a later blocking preload waits for it instead of returning early.
- Timer scheduling uses a background animation loop and `wx.CallAfter` to reduce drift and avoid UI-thread blocking.

Known tradeoffs:

- Large animated files can take longer to open because frames are decoded up front.
- High-quality scaling on very large animations is CPU-expensive.
- `None` scaling is the recommended performance mode for very large animated files.

## Build/Development Notes

Most source-level work was performed against extracted/decompiled source mirrors under `.codex_tmp/src`, then compiled back into Python 2.6 bytecode and packed into `library.zip`.

Common helper scripts retained under `.codex_tmp` include:

- Python 2.6 bytecode compile scripts.
- `replace_*_entries.py` archive replacement scripts.
- small synthetic GIF/WebP test assets.
- source mirrors used to rebuild the current patched `library.zip`.

`.codex_tmp` is useful development/debug material and can be included for repository history. To reduce storage cost while keeping it easy for another coding agent to inspect, keep it as a standard ZIP archive beside the project files. ZIP is preferred here because it is built into Windows PowerShell and can be expanded quickly with `Expand-Archive`.
